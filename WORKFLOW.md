# AI-Driven Development Lifecycle (AI-DLC) — Custom Methodology

> This file defines HOW every agent must behave on this project.
> It is not optional. It must be read before any implementation task begins.
> Applies to: Codex, Claude Code, or any AI coding agent working on this repo.

---

## Core Philosophy

You are not a code generator. You are a collaborative engineering partner.
Your job is to **understand before you build**, **clarify before you assume**, and **confirm before you commit**.

Speed without correctness is not velocity — it is rework. Ask first.

---

## The Three-Phase Protocol

Every task — no matter how small — follows three phases before a single line of code is written.

---

### PHASE 1 — DISCOVERY (Always first)

Before doing anything else:

1. Read all context documents in this order:
   - `AGENTS.md` — project identity, stack, constraints
   - `BACKLOG.md` — current phase, pending tasks, completed work
   - `WORKFLOW.md` — this file (you are reading it)
   - Relevant spec documents: `SRS.md`, `SAD.md`, `PRD.md`, `FSD.md` (if available)

2. Identify everything that is **ambiguous, unspecified, or assumption-prone** in the task at hand.

3. Output a **Discovery Report** in this format:

```
## Discovery Report

### What I understand
[Your interpretation of what needs to be built]

### What I am NOT sure about
[Numbered list — see question format below]

### Assumptions I would have made (but won't)
[List assumptions you caught yourself about to make — these become questions]
```

4. **Stop. Do not write code. Wait for answers.**

---

### PHASE 2 — CONFIRMATION (After user answers questions)

Once questions are answered:

1. Write a **Plan Summary** in this format:

```
## Implementation Plan

### Scope of this task
[Exactly what will be built — nothing more]

### Out of scope (explicitly)
[What will NOT be touched, to prevent scope creep]

### Files that will be created or modified
- [file path] — [reason]

### Key decisions made (based on your answers)
- [decision] — [rationale]

### Risks or dependencies I'm flagging
[Anything that could go wrong or depends on future phases]
```

2. **Stop. Ask: "Does this plan look correct? Should I proceed?"**

3. Wait for explicit approval — "yes", "proceed", "looks good", or equivalent.

---

### PHASE 3 — IMPLEMENTATION (Only after explicit approval)

Now build. During implementation:

- Follow Clean Architecture boundaries (Domain → Application → Infrastructure → API)
- Never introduce a new package/library without flagging it and asking first
- Never modify files outside the agreed scope without asking
- If you encounter a new ambiguity mid-implementation, **stop and ask** — do not guess
- After completing, output a **Completion Report**:

```
## Completion Report

### What was built
[Summary of what was implemented]

### Files created/modified
- [file path] — [what changed]

### What to test manually
[Steps the developer should take to verify it works]

### What this unlocks (next steps)
[What phase or task is now unblocked]

### Anything I flagged during implementation
[Surprises, concerns, or decisions that deviated from the plan]

### Steps to test the application
Add steps to run the application and do manual testing
```

## Question Format & Answer File Protocol

### How Questions Work

When the agent has clarifying questions, it does **not** ask them in chat.
Instead, it generates a file called **`QUESTIONS.md`** in the repo root and tells the developer to go fill it in.

This keeps the decision trail version-controlled, readable, and referenceable by future agents.

---

### Step 1 — Agent generates QUESTIONS.md

The agent creates `QUESTIONS.md` using this exact format for every question:

```markdown
# QUESTIONS.md — Clarifying Questions for [Task Name]
# Generated by agent on [date]
# Status: AWAITING ANSWERS
#
# Instructions for developer:
# - Read each question and its options carefully
# - Enter your chosen option letter (A, B, C, etc.) in the Answer field
# - Enter "CN" in the Answer field if clarification is needed
# - Add any context or notes in the Notes field
# - When done, save the file and prompt the agent: "Questions answered, please review QUESTIONS.md and proceed"

---

**[CATEGORY] Q1: [Short title]**
> Question: [The actual question]
> Why I'm asking: [What goes wrong if the wrong choice is made]

- A) [Option A — with brief implication]
- B) [Option B — with brief implication]
- C) [Option C — with brief implication]
- CN) Clarification needed — I'm not sure, let's discuss

Answer: __
Notes: _leave blank or add context here_

---

**[CATEGORY] Q2: [Short title]**
> Question: [The actual question]
> Why I'm asking: [What goes wrong if the wrong choice is made]

- A) [Option A — with brief implication]
- B) [Option B — with brief implication]
- C) [Option C — with brief implication]
- D) [Option D — with brief implication]
- CN) Clarification needed — I'm not sure, let's discuss

Answer: __
Notes: _leave blank or add context here_

---
```

Options are not limited to three — the agent may provide up to six options (A through F) per question depending on how many meaningful choices exist. `CN` is always the final option on every question.

---

### Step 2 — Agent notifies the developer

After creating `QUESTIONS.md`, the agent outputs this message in chat and stops:

```
📋 I've created QUESTIONS.md with [N] questions that need your input before I can proceed.

Please open QUESTIONS.md, select your answers by marking [x] next to each choice,
and add notes in the Notes field for any "Clarification needed" items.

When ready, come back and say: "Questions answered, please review QUESTIONS.md and proceed"
```

**The agent does not write any code until this prompt is received.**

---

### Step 3 — Developer fills in QUESTIONS.md

The developer opens `QUESTIONS.md` and fills in the Answer field like so:

```markdown
**[ARCHITECTURE] Q1: Hangfire Queue Strategy**
> Question: Should transcription jobs use Hangfire's default queue or a dedicated queue?
> Why I'm asking: Queue strategy affects job prioritisation and monitoring — hard to change later

- A) Use Hangfire's default queue — simpler setup, fine for now
- B) Use a dedicated "transcription" queue — better isolation and monitoring
- C) Use separate queues per job type (transcription, minutes, notifications)
- D) Use priority queues — high/medium/low
- CN) Clarification needed — I'm not sure, let's discuss

Answer: B
Notes: We may add more job types later so a dedicated queue gives us flexibility
```

For any `CN` answer, the developer adds context in the Notes field and the agent will propose a recommendation based on those notes before finalising the plan.

---

### Step 4 — Agent reads answers and proceeds

When the developer sends the proceed prompt, the agent:

1. Reads `QUESTIONS.md`
2. Extracts all `[x]` selections and Notes
3. Handles any `Clarification needed` items by proposing a recommendation based on the Notes and asking for a simple confirm/override
4. Proceeds to Phase 2 — Confirmation (Plan Summary)
5. Updates the `QUESTIONS.md` status header to `# Status: ANSWERED`, renames
   it for the task, and moves it into the appropriate delivery-cycle folder
   under `docs/archive/phaseN/` (does not delete it)

---

### Question Categories
| Tag | Use for |
|-----|---------|
| `[ARCHITECTURE]` | Structural decisions, layer boundaries, patterns |
| `[DATA MODEL]` | Entity shape, relationships, nullable fields, enums |
| `[FEATURE]` | Scope, behaviour, edge cases, business rules |
| `[TECH CHOICE]` | Library selection, package versions, tooling |
| `[INTEGRATION]` | How this connects to another service, API, or phase |
| `[ERROR HANDLING]` | What happens on failure, retry logic, user feedback |
| `[SCOPE]` | Is X in or out of this task / phase |
| `[SECURITY]` | Auth, permissions, data exposure concerns |

---

### Naming Convention and location for QUESTIONS.md archives

Once answered, the agent renames and moves the file before the next task
generates a new active `QUESTIONS.md`:

```
docs/archive/phase1/QUESTIONS_phase4_hangfire.md
docs/archive/phase1/QUESTIONS_phase5_transcription.md
QUESTIONS.md                     ← active/current file only
```

Use the archive folder for the current delivery cycle (`phase1`, `phase2`, and
so on). Never move an unanswered active `QUESTIONS.md` into the archive. This
keeps a complete decision log without filling the repository root with closed
records.

## Ambiguity Checklist

Run this checklist mentally before EVERY task. If any item is unchecked, it becomes a question.

### Data & Domain
- [ ] Is the entity / data model fully defined for this feature?
- [ ] Are all required vs optional fields specified?
- [ ] Are enum values or status codes explicitly listed?
- [ ] Are relationships (one-to-many, etc.) clear?

### Feature Behaviour
- [ ] Are success and failure paths both described?
- [ ] Are edge cases and boundary conditions handled in the spec?
- [ ] Is pagination, sorting, or filtering required?
- [ ] Are there performance or concurrency requirements?

### Architecture & Code
- [ ] Is the correct Clean Architecture layer clear for each piece of work?
- [ ] Does this feature introduce a new cross-cutting concern (logging, caching, validation)?
- [ ] Will this touch existing code that could break other features?
- [ ] Are there existing patterns in the codebase I should follow?

### Integration Points
- [ ] Does this depend on a phase or feature not yet built?
- [ ] Are external service contracts (OpenAI, Whisper, FFmpeg) fully specified?
- [ ] Is the Hangfire job interface (input, output, failure behaviour) defined?

### Security & Auth
- [ ] Does this endpoint/feature require authentication?
- [ ] Are there role/permission checks needed?
- [ ] Is any sensitive data (audio files, transcripts) being exposed?

## MeetingMind AI — Project-Specific Rules

These rules are permanent and apply on top of the general protocol.

### Stack Constraints
- **Backend**: ASP.NET Core Web API — Clean Architecture (Domain / Application / Infrastructure / API layers)
- **Database**: PostgreSQL via EF Core — always use migrations, never modify DB directly
- **Background Jobs**: Hangfire — all long-running work (transcription, minutes generation) runs as Hangfire jobs
- **File Storage**: Local filesystem (configurable path) — no cloud storage unless explicitly decided
- **AI Services**: OpenAI Whisper (transcription) + GPT (minutes generation) — always use interfaces, never call OpenAI SDK directly from a job
- **Frontend**: React — coordinate with backend API contracts before building UI

### Naming & Structure Conventions
- Follow existing entity naming in `Domain/Entities/`
- Application layer: Commands and Queries via MediatR (CQRS pattern)
- Infrastructure layer: EF Core repositories + external service implementations
- API layer: Controllers thin — delegate everything to MediatR handlers

### Phase Discipline
- Never implement work from a future phase unless explicitly asked
- Always check `BACKLOG.md` to confirm what phase is active before starting
- Mark tasks as complete in `BACKLOG.md` after the Completion Report

### Hangfire-Specific Rules (Phase 4+)
- Every job must have: an interface in Application layer, implementation in Infrastructure layer
- Jobs must log start, progress, and completion
- Jobs must handle failure gracefully — update meeting status to `Failed` with error detail
- Never hard-code retry counts — use configuration
- Ask before adding any new Hangfire queue or job type not in the backlog

### OpenAI Integration Rules
- Never expose API key in code — always read from configuration/environment
- Always wrap OpenAI calls in try/catch with meaningful error messages
- Whisper and GPT calls must be in separate service classes
- If a call fails, the Hangfire job must update the meeting status and not silently swallow the error

## What "Done" Means

A task is only done when:

1. Code compiles with no errors
2. EF Core migrations (if any) are generated and applied
3. Completion Report is written
4. `BACKLOG.md` is updated to reflect task status
5. Developer has been told exactly what to test manually

---

## Anti-Patterns — Never Do These

| Anti-pattern | Why it's banned |
|---|---|
| Assume and implement | Creates rework, breaks architecture |
| Add packages without asking | Breaks consistency, may conflict |
| Modify out-of-scope files | Causes unexpected breakage |
| Skip the Plan Summary | Developer loses visibility |
| Implement multiple phases at once | Violates phase discipline |
| Guess at data model fields | Schema changes are expensive |
| Silently swallow exceptions | Hides bugs in production |
| Write business logic in Controllers | Violates Clean Architecture |
| Call OpenAI SDK directly from jobs | Breaks testability |
